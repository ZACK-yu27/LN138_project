## 一、项目基本信息

- **项目类型**：必须是一个**功能完整的应用程序**，本项目采用 **Web App（Flask + 原生 HTML/CSS/JS）** 形式实现。
- **核心方向**：推荐以**数据处理、分析与可视化**为主题（对应课件 `课件/14-Matplotlib数据可视化.pdf`），结合文件操作、数据结构、函数、类与 Matplotlib 可视化。
- **禁止方向**：不建议使用课程未涉及的机器学习库（如 `scikit-learn`、`tensorflow`）。项目必须与课程内容匹配，重在扎实运用所学。
- **原创性要求**：严禁抄袭。项目将进行代码查重，核心代码大量直接复制自网络或他人，相关评分项记为不及格。

## 二、选题要求

1. **必须是功能完整的应用程序**，不能是零散脚本或无交互的代码片段。
2. **推荐选题方向**（优先选择）：
   - 围绕某个真实或模拟数据集，完成“读取 → 清洗 → 分析 → 可视化 → 结论”的完整流程。
   - 数据来源可为公开 CSV、本地调研数据、模拟生成数据等。
   - 分析维度应有意义，避免仅做简单求和或计数。
3. **技术匹配度**：选题必须能够自然覆盖课程核心知识模块中的至少 **4 项**（详见下文“技术运用要求”）。
4. **创新性**：鼓励从新问题角度、技术组合方式、数据分析视角、可视化呈现方式等方面体现创新。

## 三、项目要求

### 3.1 功能要求

- 使用文件操作读取数据文件（推荐 CSV 格式）。
- 使用列表、字典、元组、集合等数据结构组织和处理数据。
- 使用函数对功能进行模块化封装。
- 使用类与面向对象编程组织至少 1 个有实际意义的类。
- 使用 Matplotlib 绘制常见图表（折线图、柱状图、散点图、饼图等至少 2 种）。
- 包含异常处理，确保文件读取、用户输入等关键环节不会因异常直接崩溃。
- 程序应能在命令行或其他简单界面中完整运行，并给出清晰输出。

### 3.2 代码要求

- 代码必须分文件组织，禁止全部堆在单一 `.py` 文件中。
- 命名规范（模块名、类名、函数名、变量名）符合 PEP 8。
- 每个函数/类必须有清晰的注释或文档字符串，说明功能、参数、返回值。
- 关键逻辑必须有中文或英文注释。
- 无明显 Bug，运行稳定。

## 六、AI Agent 工作规范（必读）

> 本规范适用于所有组员在本地拉取仓库后，由其 AI Agent 协助开发时的行为约束。AI Agent 在生成或修改任何代码前，必须先阅读本 README 和项目报告要求，确保输出符合满分标准。

1. **先读规范再动手**：每次会话开始时，AI Agent 必须先读取 `README.md` 和当前项目结构，确认自己所负责的模块。

   - 严禁从网络直接复制核心代码片段。
   - 严禁生成无法运行的“伪代码”或占位函数。
3. **代码风格**：遵循 PEP 8，使用中文或英文注释，函数/类必须有文档字符串。
4. **最小改动原则**：只修改自己负责的模块，不擅自改动他人模块；如需调整接口，必须先与组长和相关组员沟通。
5. **自测要求**：生成代码后，AI Agent 必须尝试运行 `python main.py` 或相关测试命令，确认无语法错误和明显逻辑错误。

### 6.2 按角色的 Agent 分工规范（3 人 Web App 项目）

> 本项目为 3 人小组，采用 Flask + SQLite + 原生 HTML/CSS/JS 开发饮食记录 & 推荐系统。原 4 人角色中的“主程序整合”工作由组长兼任。

#### 角色 A：数据采集与数据库

- **负责文件**：`data_loader.py`、`db.py`、`data/*.csv`
- **必须展示的技术点**：
  - 字符串处理（第 2 章）：CSV 解析、字段清洗、类别名称匹配。
  - 文件操作（第 7 章）：使用 `with open(...)` 读取食堂/门店 CSV。
  - 数据库应用（第 10 章）：SQLite 表创建、CRUD 操作。
  - 异常处理（第 9 章）：处理文件不存在、编码错误、数据缺失等。
  - 列表/字典（第 3、5 章）：将 CSV 数据组织为字典列表，写入数据库。
- **输出要求**：
  - 提供 `load_cafeteria_data()`、`load_shop_data()` 等加载函数。
  - 提供 `db.py` 中数据库初始化、用户表、饮食记录表、食物分类表的 CRUD 函数。

#### 角色 B：后端核心逻辑与 API

- **负责文件**：`app.py`、`analyzer.py`、`recommender.py`
- **必须展示的技术点**：
  - 函数（第 6 章）：封装营养评分、预算分析、推荐算法。
  - 类与 OOP（第 8 章）：定义 `DietAnalyzer`、`Recommender` 等类。
  - 控制结构（第 4 章）：循环、条件筛选、推荐规则判断。
  - 列表/字典（第 3、5 章）：列表推导式、字典统计、推荐排序。
  - 数据分析与可视化（第 11、12、13 章）：营养结构评分、预算趋势计算。
- **输出要求**：
  - `app.py` 提供 Flask API 接口（见 `PRD.md` 第 8 节）。
  - `analyzer.py` 实现营养评分、预算分析、健康目标匹配。
  - `recommender.py` 实现“去哪吃”“什么时候吃”“吃什么”推荐。

#### 角色 C：前端、可视化与 PDF 导出（组长兼任整合）

- **负责文件**：`templates/*.html`、`static/css/style.css`、`static/js/app.js`、`visualizer.py`、`pdf_exporter.py`
- **必须展示的技术点**：
  - Matplotlib（第 13 章）：绘制至少 4 种图表（折线、柱状、饼图、水平条形图）。
  - 函数（第 6 章）：每个图表一个函数，PDF 导出函数。
  - 字符串处理（第 2 章）：图表标题、标签格式化。
  - 文件操作（第 7 章）：图表图片保存、PDF 生成。
- **输出要求**：
  - 前端页面：信息录入页、饮食记录页、数据看板页、推荐页。
  - `visualizer.py` 函数接受后端数据，生成并保存图表到 `output/`。
  - `pdf_exporter.py` 将看板图表整合为 PDF 报告。
  - 编写 `requirements.txt`、运行说明，负责项目整合与报告撰写。

## 七、Git 使用规则

为确保多人协作不冲突、版本清晰可追溯，所有组员及其 AI Agent 必须遵守以下 Git 规则。

### 7.1 分支策略

- **主分支**：`main` 为保护分支，仅存放可运行、稳定的代码。
- **个人开发分支**：每位组员在本地创建自己的功能分支，命名规则：
  - `feature/角色-模块名-姓名拼音`，例如：
    - `feature/data-db-zhangsan`      # 数据采集与数据库
    - `feature/backend-api-lisi`      # 后端逻辑与 API
    - `feature/frontend-pdf-wangwu`   # 前端与 PDF 导出
- **禁止**：直接在 `main` 分支上开发或提交。

### 7.2 提交（commit）规范

使用 **Conventional Commits** 风格，便于自动生成变更日志和回顾：

```text
<类型>(<模块>): <简短描述>

<可选：详细说明>
```

常用类型：

| 类型 | 含义 | 示例 |
|:----|:----|:----|
| `feat` | 新功能 | `feat(loader): 添加 CSV 读取与缺失值处理` |
| `fix` | 修复 Bug | `fix(analyzer): 修复中位数计算边界问题` |
| `docs` | 文档/报告 | `docs(report): 完善技术实现章节` |
| `style` | 代码格式 | `style(visualizer): 统一图表标题格式` |
| `refactor` | 重构 | `refactor(main): 优化主程序流程` |
| `test` | 测试相关 | `test(analyzer): 添加统计函数单元测试` |
| `chore` | 杂项/配置 | `chore: 更新 requirements.txt` |

### 7.3 协作流程

1. **开始工作前**：
   ```bash
   git checkout main
   git pull origin main
   git checkout -b feature/你的分支名
   ```

2. **开发过程中**：
   - 小步提交，每个 commit 只做一件事。
   - commit 前自测代码能否运行。

3. **推送到远程**：
   ```bash
   git add .
   git commit -m "feat(模块): 描述"
   git push origin feature/你的分支名
   ```

4. **合并到 main**：
   - 由组长或相关组员在 GitHub 发起 **Pull Request**。
   - PR 标题与 commit 风格一致。
   - 合并前必须确认：
     - [ ] 与最新 `main` 无冲突
     - [ ] 代码能运行
     - [ ] 不破坏他人模块
   - 由组长审核后合并。

5. **冲突解决**：
   - 优先在本地解决：
     ```bash
     git checkout feature/你的分支名
     git pull origin main
     # 解决冲突后
     git add .
     git commit -m "merge: 解决与 main 的冲突"
     git push origin feature/你的分支名
     ```

### 7.4 禁止事项

- 禁止 `--force` 推送（`git push -f`）。
- 禁止提交大文件（如超过 10MB 的数据集可放网盘链接，仓库只放示例小数据）。
- 禁止提交临时文件、IDE 配置、虚拟环境文件夹（应提前配置 `.gitignore`）。
- 禁止在 commit message 中使用无意义描述，如“修改”、“1”、“test”。
