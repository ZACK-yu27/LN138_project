# 饮食记录 & 推荐系统 — 产品需求文档（PRD）

> **项目性质**：中山大学《程序设计Ⅳ（实验）》期末编程项目  
> **团队规模**：3 人  
> **开发周期**：2 周  
> **技术栈**：Python + Flask + SQLite + 原生 HTML/CSS/JS + Matplotlib

---

## 1. 项目背景与目标

### 1.1 背景
大学生在日常饮食中面临三大痛点：
1. **选择困难**：每天不知道吃什么，尤其到了饭点频繁纠结。
2. **健康与预算失控**：不清楚自己每月在饮食上花了多少钱，也不清楚营养结构是否均衡。
3. **就餐时机不佳**：去食堂时常常赶上高峰期，排队时间长；或错过低谷期优惠/空闲时段。

### 1.2 目标
开发一个**功能完整的 Web 应用程序**，帮助大学生：
- 记录每日饮食，自动分析营养结构与预算执行。
- 根据个人健康目标（减肥 / 维持身材 / 健身增重）获得健康评分。
- 获得“去哪吃”和“什么时候吃”的个性化推荐。
- 将数据看板以 PDF 格式导出，便于查看和分享。

## 2. 用户画像与使用场景

### 2.1 用户画像
- **主要用户**：在校大学生。
- **使用设备**：个人电脑或校园内可访问的本地服务器。
- **技术能力**：无需编程基础，通过浏览器即可使用。

### 2.2 使用场景

| 场景 | 用户需求 | 系统行为 |
|:---|:---|:---|
| 首次使用 | 设置个人预算和健康目标 | 打开信息录入页，保存用户配置 |

| 控制预算 | 查看本月餐饮支出 | 看板展示预算使用进度与趋势 |
| 饭点决策 | 不知道去哪吃、什么时候去 | 推荐食堂/门店、推荐低谷期时段 |
| 期末总结 | 导出饮食数据报告 | 一键下载 PDF 看板报告 |

## 3. 功能需求

### 3.1 信息采集（首次使用）

用户首次打开系统时，需要填写以下信息：

| 字段 | 类型 | 必填 | 说明 |
|:---|:---|:---|:---|
| 用户名 | str | 是 | 用于区分本地用户 |
| 月度餐饮预算 | float | 是 | 单位：元 |
| 健康目标 | enum | 是 | 选项：`减肥` / `维持身材` / `健身增重` |
| 饮食偏好 | multi-select | 否 | 如：偏辣、清淡、少油、素食等 |

**业务规则**：
- 健康目标一旦选择，系统按预设标准评分，用户不可自定义权重，但可切换目标。
- 预算为自然月周期，每月 1 日自动重置（或用户手动重置）。

### 3.2 饮食记录录入

#### 3.2.1 录入字段

| 字段 | 类型 | 必填 | 说明 |
|:---|:---|:---|:---|
| 日期 | date | 是 | 默认当天，可修改 |
| 餐次 | enum | 是 | `早餐` / `午餐` / `晚餐` / `加餐` |
| 就餐地点 | enum | 是 | `食堂` / `外卖` / `校外门店` |
| 具体档口/门店 | str | 否 | 如：一食堂二楼、麦当劳 |
| 食物名称 | str | 是 | 如：黄焖鸡米饭 |
| 价格 | float | 是 | 单位：元 |
| 食物分类 | multi-select | 是 | 主食 / 蔬菜 / 肉类 / 水果 / 饮料 / 零食 / 其他 |
| 各类别分量 | int | 是 | 以“拳头大小”为单位，每个类别 0–5 拳 |
| 是否外卖 | bool | 是 | 用于统计外卖占比 |

#### 3.2.2 分量量化规则

- 采用“拳头大小”作为直观量化单位。
- 每类食物独立填写分量：
  - 0 拳：未食用
  - 1 拳：少量
  - 2–3 拳：正常
  - 4–5 拳：过量
- 系统将拳头数映射为相对热量系数，用于营养评分。

#### 3.2.3 录入示例

```text
日期：2026-06-12
餐次：午餐
就餐地点：食堂
具体档口：一食堂二楼
食物名称：黄焖鸡米饭
价格：18.0
食物分类：主食 2 拳、肉类 1 拳、蔬菜 0.5 拳
是否外卖：否
```

### 3.3 数据看板

看板是系统的核心页面，包含以下模块：

#### 3.3.1 饮食统计

- 本月总餐次、总支出、日均支出。
- 各餐次（早/午/晚/加餐）支出占比。
- 外卖占比。
- 最常去的食堂/门店 TOP 5。

#### 3.3.2 预算控制

- 本月已用预算 vs 月度预算。
- 剩余预算。
- 预算使用进度条。
- 按当前日均消费预测本月是否会超支。

#### 3.3.3 健康水平分析

- 营养结构评分（0–100 分）。
- 实际摄入结构 vs 健康目标建议结构（柱状图/雷达图）。
- 健康建议：如“蔬菜摄入不足”“主食偏多”“外卖占比过高”。
- 日均热量估算趋势。

#### 3.3.4 就餐推荐

- **去哪吃**：基于预算、位置、历史偏好推荐食堂档口或校外门店。
- **什么时候吃**：结合食堂高峰期/低谷期数据，推荐最佳就餐时间。
- **吃什么**：基于历史偏好和营养缺口推荐餐品组合。

### 3.4 就餐推荐

#### 3.4.1 推荐维度

| 推荐类型 | 输入 | 输出 |
|:---|:---|:---|
| 去哪吃 | 当前位置、预算、偏好 | 推荐 3 个食堂档口或门店，附理由 |
| 什么时候吃 | 当前时间、目标食堂 | 推荐低谷期时段，避开排队 |
| 吃什么 | 餐次、预算、健康目标 | 推荐 3 个餐品组合，附营养说明 |

#### 3.4.2 推荐规则

- **预算内推荐**：过滤掉价格高于当前剩余日均预算的选项。
- **营养缺口推荐**：如果某类食物摄入不足，优先推荐含该类的餐品。
- **高峰期规避**：午餐 11:45–12:30 为高峰期，推荐 11:30 前或 12:30 后。
- **历史偏好**：优先推荐用户过去消费频次高、评分好的选项。

### 3.5 图表 PDF 导出

- 看板页面提供“导出 PDF”按钮。
- 导出的 PDF 包含：
  - 饮食统计摘要
  - 预算控制图表
  - 健康水平分析图表
  - 推荐结果
- 使用 Matplotlib 的 `PdfPages` 后端生成多页 PDF。

## 6. 推荐算法与评分规则

### 6.1 营养结构评分

#### 6.1.1 健康目标与理想结构

| 健康目标 | 主食 | 蔬菜 | 肉类 | 水果 | 饮料/零食 | 其他 |
|:---|:---|:---|:---|:---|:---|:---|
| 减肥 | 30% | 30% | 20% | 10% | 5% | 5% |
| 维持身材 | 35% | 25% | 20% | 10% | 5% | 5% |
| 健身增重 | 40% | 15% | 30% | 5% | 5% | 5% |

#### 6.1.2 评分计算

1. 统计每个类别的拳头总数。
2. 计算实际占比。
3. 计算每个类别与理想占比的绝对偏差。
4. 总偏差越小，得分越高：

```
score = max(0, 100 - 50 × total_deviation)
```

5. 根据偏差生成建议，如“蔬菜摄入不足”“主食偏多”。

### 6.2 健康目标匹配度

- 根据用户选择的健康目标，使用对应的营养结构标准。
- 用户不可自定义权重，只能切换目标后重新计算。

### 6.3 食堂高峰期/低谷期建议

- 高峰期：每个档口独立配置 `peak_start`、`peak_end`。
- 系统根据当前时间判断目标食堂是否处于高峰：
  - 若在高峰前 15 分钟内：建议“现在就去，避开高峰”。
  - 若在高峰期内：建议“错峰，12:30 后再去”。
  - 若在高峰期后：建议“当前为低谷期，可前往”。

### 6.4 预算内推荐

- 计算剩余日均预算：`剩余预算 / 本月剩余天数`。
- 推荐价格 ≤ 剩余日均预算的餐品。
- 若预算紧张，优先推荐食堂低价档口。

## 9. 非功能需求

### 9.1 性能
- 系统为本地部署，并发量低，响应时间 < 1s。
- 数据库查询简单，无需复杂索引优化。

### 9.2 可维护性
- 代码分模块组织，符合 PEP 8。
- 函数与类必须有文档字符串。
- API 返回统一 JSON 格式。

### 9.3 兼容性
- 后端支持 Python 3.10+。
- 前端兼容主流浏览器（Chrome、Edge、Firefox）。

### 9.4 数据安全
- 用户数据仅存储在本地 SQLite，不上传服务器。
- 不涉及真实身份敏感信息。
