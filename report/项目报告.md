## 一、摘要

本项目针对大学生日常饮食中的三大痛点——"选择困难""健康与预算失控""就餐时机不佳"，开发了一个功能完整的 Web 应用程序。系统支持用户记录每日饮食、自动分析营养结构与预算执行情况，并基于个人健康目标（减肥/维持/健身增重）提供"去哪吃""什么时候吃""吃什么"三类个性化推荐。数据看板以 Matplotlib 可视化图表呈现，支持一键导出 PDF 报告。项目创新性地采用"拳头大小"作为直观量化单位，将复杂营养学简化为可操作的健康评分，同时将像素风视觉设计融入数据分析工具，在课程要求的 Python 基础技术展示之外，体现了技术与设计的交叉创新。

## 二、项目设计

### 2.1 模块划分

项目采用经典的三层架构：

| 层级 | 文件 | 职责 |
|------|------|------|
| 数据层 | `db.py`, `data_loader.py` | SQLite 数据库操作、CSV 数据读取与清洗 |
| 逻辑层 | `analyzer.py`, `recommender.py` | 营养评分、预算分析、三类推荐算法 |
| 展示层 | `app.py`, `visualizer.py`, `pdf_exporter.py` | Flask API、图表生成、PDF 导出 |
| 前端层 | `templates/*.html`, `static/css/style.css`, `static/js/app.js` | 4 个页面、像素风样式、交互逻辑 |

### 2.2 核心数据结构

**用户配置表（users）**：存储用户名、月度预算、健康目标（减肥/维持/健身增重）、饮食偏好。

**饮食记录主表（diet_records）**：记录日期、餐次、就餐地点、食物名称、价格、是否外卖。通过外键关联用户。

**记录分类表（record_items）**：将饮食记录的食物分类与分量独立存储，支持一餐多类（如主食+肉类+蔬菜）。

**食堂/门店数据表**：存储外部菜品数据，包含高峰期时间、平均等待时间等推荐所需信息。

### 2.3 推荐算法设计

营养评分公式：`score = max(0, 100 - 50 * total_deviation)`，其中 total_deviation 为各类别实际占比与理想占比的绝对偏差之和。三类推荐均基于"预算过滤+营养缺口+历史偏好+高峰期规避"的多因子综合评分模型。

### 3.2 列表、元组、字典、集合（第3、5章）

`analyzer.py` 使用 `defaultdict` 和 `Counter` 进行字典统计：

```python

for record in records:
    for item in record.get('items', []):
```

```python
only_canteen = canteen_dates - takeout_dates
mixed_dates = canteen_dates & takeout_dates
```

### 3.3 控制结构与异常处理（第4、9章）

`db.py` 中对文件不存在和数据库异常进行捕获：

```python
try:
    with open(file_path, "r", encoding="utf-8") as f:
        # ...
```

## 四、独创性声明

1. **"拳头大小"量化体系**：将抽象的营养学概念转化为大学生可直观理解的"拳头"单位，每类食物 0-5 拳的分量设计既简化了数据录入，又保证了营养分析的合理性。

2. **像素风与数据可视化的融合**：在数据分析工具中引入像素风（pixel art）视觉设计，采用直角边框、块状阴影、低饱和暖色调，使学术工具呈现出独特的游戏化视觉体验，增强了用户使用的愉悦感。

3. **多因子推荐模型**："去哪吃/什么时候吃/吃什么"三类推荐综合考虑预算、营养缺口、历史偏好、高峰期规避、距离等维度，而非简单的价格排序，体现了课程知识在真实场景中的创造性组合。

4. **Web 形式的完整应用**：不同于命令行程序，本项目采用 Flask + 原生前端构建了可交互的 Web 应用，在课程技术要求框架内实现了较高的工程完整性。

### 6.2 协作流程

- **版本控制**：使用 Git 进行协作，遵循 Conventional Commits 规范（`feat`/`fix`/`docs`/`refactor` 等类型）。每个成员在自己的功能分支上开发，完成后通过 Pull Request 由组长审核合并到 `main` 分支。
- **沟通机制**：三人小组在线协作，各模块接口（数据库方法签名、API 路由格式）在开发前由组长统一设计并写入 `PRD.md`，三方对接口达成一致后再并行开发，避免整合冲突。
- **模块整合**：组长负责将 A/B/C 三个角色的代码整合为完整可运行的 Web 应用。整合策略为"接口先行、模块独立、逐层集成"——先确定数据库 CRUD 接口，再实现后端分析逻辑，最后接入前端展示层。
- **质量保障**：各组员完成自己模块后，运行 `python app.py` 进行功能验证。组长在合并前检查代码能正常运行且不破坏其他模块。新增 `test_core.py` 对核心分析函数进行自动化测试，确保重构不影响已有功能。
